## 核心问题：PC 预训练 vs 纯手机端 RL {#核心问题}

### 能跳过 PC 训练，直接在手机上 RL 吗？

**答案：可以。不仅可行，而且更有意思。**

| 指标 | 估计值 |
|--------|----------|
| DOOM 帧率 | 35 fps |
| CNN-DQN 推理 | 3-5 ms（TFLite GPU 加速） |
| 训练步（每 4 帧） | 25-30 ms（反向传播） |
| 有效游戏+训练速度 | ~30 fps |
| 经验回放缓冲区 | 50K 条经验（~1 MB 内存） |
| 模型大小 | 2-5 MB（TFLite 量化） |
| 达到"不太蠢"所需帧数 | 50K-100K |
| **达到基本能力的时间** | **~30-60 分钟** |
| 达到"还不错"所需帧数 | 500K-1M |
| **达到不错水平的时间** | **~5-10 小时** |
| 达到"很好"所需帧数 | 5M+ |
| **达到高水平的时间** | **~50+ 小时** |

### 权衡对比：预训练 vs 纯手机端

| | PC 预训练 + 部署 | 纯手机端 RL |
|---|---|---|
| **首次可用** | 立即（预训练模型） | ~30-60 分钟 |
| **能力上限** | 训练时即固定 | 持续提升 |
| **电池影响** | 仅推理（~3%/小时） | 训练耗电更多（~8%/小时） |
| **用户体验** | 开箱即用 | "看它学习"更有参与感 |
| **个性化** | 通用玩法风格 | 适应你自己的风格 |
| **开发难度** | 高（需 ViZDoom、PC GPU） | 中（全 Java+TFLite） |
| **模型可移植性** | 任何手机立即可用 | 每台手机独立学习 |
| **酷炫程度** | 😎 | 🤯 |

**推荐**：纯手机端 RL，可选预训练种子模型作为起点。两全其美。

## 方案 1：DQN 系列 {#方案1-dqn}

DQN 学习函数 Q(s,a)，估算在状态 s 下采取动作 a 的期望未来奖励。智能体选择 Q 值最高的动作。

### 1a. 基础 DQN

```
输入：  4 帧堆叠（120×68×4）→ 捕捉运动信息
        8 个游戏状态值（生命值、弹药×4、护甲、击杀数、武器）

CNN：   Conv(32, 8×8, stride=4) → ReLU
        Conv(64, 4×4, stride=2) → ReLU
        Conv(64, 3×3, stride=1) → ReLU
        Flatten(3136) + StateConcat(+8) → 3144 单元
全连接： Dense(512) → ReLU
        Dense(224) → 224 个动作的 Q 值

大小：   ~2 MB（INT8 量化）
推理：   ~3-5 ms（TFLite GPU）
```

**优点**：简单、Atari 验证有效、模型小
**缺点**：高估 Q 值、收敛较慢

### 1b. Double DQN

使用两个网络：一个选择动作（在线网络），一个评估动作（目标网络）。减少高估偏差。

**优点**：更稳定、最终性能更好
**缺点**：内存稍大（两个模型）、推理速度相同

### 1c. Dueling DQN

将 Q(s,a) 拆分为 V(s) + A(s,a) — 状态价值 + 动作优势。学习哪些状态本身就很有价值。

```
CNN → ┌─ V(s) 流  ─┐
      │             ├→ Q(s,a) = V(s) + A(s,a) - mean(A)
      └─ A(s,a) 流 ┘
```

**优点**：多动作价值相近时表现更好、学习更快
**缺点**：网络稍大

### 1d. Rainbow DQN

组合 6 项改进：Double DQN + Dueling + 优先回放 + 多步 + 分布式 + Noisy Nets。

**优点**：DQN 最高水平
**缺点**：~10 MB 模型、实现复杂、训练成本高

### 1e. DRQN（深度循环 Q 网络）

添加 LSTM 层处理部分可观测性和时序依赖。

```
CNN → LSTM(256) → FC → Q 值
```

**优点**：记住过往帧（对暗室、开门场景有用）
**缺点**：~5 MB 模型、<10ms 推理、需要序列批次训练

### DQN 变体总结

| 变体 | 大小 | 推理 | 训练 | DOOM 适用性 |
|---------|------|-----------|----------|-----------------|
| 基础 DQN | 2 MB | 3-5ms | 简单 | ⭐⭐⭐ 好基线 |
| Double DQN | 2.5 MB | 3-5ms | 简单 | ⭐⭐⭐⭐ 更稳定 |
| Dueling DQN | 3 MB | 4-6ms | 简单 | ⭐⭐⭐⭐ 更适合 DOOM |
| Double+Dueling | 3.5 MB | 4-6ms | 简单 | ⭐⭐⭐⭐⭐ 推荐 |
| Rainbow | 10 MB | 8-12ms | 复杂 | ⭐⭐⭐ 最强但重 |
| DRQN | 5 MB | 8-10ms | 中等 | ⭐⭐⭐⭐ 迷宫关卡 |

## 方案 2：策略梯度方法（PPO、A2C）{#方案2-ppo}

不学 Q 值再选最大值，而是直接学习策略 π(a|s) — 动作的概率分布。

### 2a. PPO（近端策略优化）

业界标准的连续控制和游戏 AI 方法。使用 Actor（策略）+ Critic（价值）架构。

```
Actor:  CNN → FC(256) → FC(224) → softmax → 动作概率
Critic: CNN → FC(256) → FC(1)    → 状态价值

总计：   ~8 MB
推理：   ~10-15 ms（TFLite）
训练：   ~40-50 ms 每次更新
```

**优点**：
- 样本效率比 DQN 更高
- 自然处理随机策略
- 可输出动作概率（对探索有用）
- OpenAI Five 用 PPO 打 Dota 2

**缺点**：
- 需要 on-policy 数据（不能随意复用旧经验）
- 模型更大、推理更慢
- 超参数更多
- GAE（广义优势估计）增加复杂度

### 2b. A2C / A3C

比 PPO 简单，使用优势函数 A(s,a) = Q(s,a) - V(s) 减少方差。

**优点**：比 PPO 轻量、适合并行训练
**缺点**：稳定性较差、A3C 需要多线程

### PPO vs DQN 在 DOOM 上

| | DQN | PPO |
|---|---|---|
| 样本效率 | 中 | **更高** |
| 稳定性 | 中 | **更高** |
| 模型大小 | **2-5 MB** | 8-12 MB |
| 推理速度 | **3-5 ms** | 10-15 ms |
| 实现复杂度 | **低** | 高 |
| 手机端训练 | **更容易** | 更难（需要 on-policy 缓冲区） |
| DOOM 基准测试 | 良好 | **ViZDoom 中更好** |

## 方案 3：世界模型（DreamerV3）{#方案3-世界模型}

不直接学 Q 值或策略，而是学习游戏世界的压缩内部表示，然后在该表示中规划或学习策略。

```
┌─────────────────────────────────────────────────────┐
│  世界模型                                            │
│                                                      │
│  帧 ──→ 编码器 ──→ z_t（32 维潜变量）                │
│                        │                             │
│              ┌─────────▼──────────┐                  │
│              │ RSSM（循环 SSM）   │                  │
│              │ h_t = f(h_t-1, z_t-1, a_t-1)        │
│              │ 预测：z_t, 奖励, 是否继续            │
│              └─────────┬──────────┘                  │
│                        │                             │
│              ┌─────────▼──────────┐                  │
│              │ Actor-Critic       │                  │
│              │ π(a|h_t)   V(h_t)  │                  │
│              │ 在潜空间中训练     │                  │
│              └────────────────────┘                  │
└─────────────────────────────────────────────────────┘
```

**DreamerV3 数据**：
- 模型大小：~20-50 MB（手机太大）
- 推理：~50-100ms/步（35fps 太慢）
- 训练：需要 GPU（不适合手机）
- 在 Minecraft、Atari 上效果出色

**结论**：❌ v1.0 不可行。等 2-3 年后骁龙 NPU 成熟 + 模型压缩再看。

## 方案 4：VLM 作为游戏 AI {#方案4-vlm}

### 为什么 VLM 无法玩实时 DOOM

| | CNN-DQN | SmolVLM 2B（手机端） | GPT-4V（云端） |
|---|---|---|---|
| 模型大小 | 2-5 MB | 2-4 GB | 100+ GB |
| 每帧推理 | 3-5 ms | 500-2000 ms | 1000-5000 ms |
| 可达帧率 | 30-35 | <1 | <1 |
| 手机可运行 | ✅ | ⚠️ 仅骁龙 NPU | ❌ 需联网 |
| 输出格式 | 动作向量 | 文本→解析 | 文本→解析 |
| 延迟（云端 RTT） | 0 | 0 | 50-300ms |

DOOM 需要 28ms/帧才能达到 35fps。即使最小的手机端 VLM 也慢了 **100-400 倍**。

### VLM 可以如何参与（v2.0 混合架构）

```
┌─────────────────────────────────────────────────┐
│  快速层（手机端 CNN-DQN，每帧运行）              │
│  • 移动、瞄准、射击、躲避                        │
│  • 30 fps、2 MB 模型                            │
├─────────────────────────────────────────────────┤
│  慢速层（可选云端/本地 VLM，0.2 Hz）             │
│  • 每 2-5 秒一次战略决策                         │
│  • "3 个 cacodemon 在前 → 换等离子枪"            │
│  • "秘密墙壁纹理 → 尝试打开"                      │
│  • "生命值 < 25 → 找最近的医疗包"                 │
│  • "需要蓝钥匙 → 向东走向门"                      │
└─────────────────────────────────────────────────┘
```

**VLM 战略提供者**（v2.0 选项）：

| 模型 | 大小 | 推理 | 质量 | 备注 |
|-------|------|-----------|---------|-------|
| GPT-4V (API) | 云端 | 1-3s | ⭐⭐⭐⭐⭐ | 最佳战略、需联网 |
| Claude 3.5 Sonnet (API) | 云端 | 1-3s | ⭐⭐⭐⭐⭐ | 推理能力强 |
| Qwen2-VL 2B | 2 GB | 500-800ms | ⭐⭐⭐ | 骁龙 NPU 手机端 |
| SmolVLM 256M | 500 MB | 200-400ms | ⭐⭐ | 勉强可用 |
| Phi-3.5-Vision 4B | 8 GB | 1-2s | ⭐⭐⭐⭐ | 手机太大 |

### VLA（视觉-语言-动作模型）

如 Google 的 RT-2，直接从视觉+语言输出机器人动作。

**结论**：❌ 不适合 DOOM。VLA 为物理机器人操作设计，不适用于实时 FPS。

### WAM（世界动作模型）

结合世界模型与动作预测的架构。最接近 DreamerV3 加动作输出头。

**结论**：与 DreamerV3 同样问题 — 太大太慢。

## 方案 5：混合架构 {#方案5-混合}

### 5a. CNN-DQN + VLM 策略层

DQN 处理实时战斗，VLM 提供偶尔的战略指导。

**优点**：两全其美
**缺点**：VLM 层增加复杂度、需联网或大模型

### 5b. CNN-DQN + 硬编码启发式规则

用简单规则过滤 DQN 的输出：

```
规则示例：
• 生命值<20% → 优先找医疗包
• 无弹药 → 强制切换武器
• 靠近门 → 提升 USE 动作权重
```

**优点**：简单、无需额外模型
**缺点**：规则脆弱、不通用

### 5c. 混合专家（小型专家模型）

训练 4-5 个 0.5MB 的微型专家 DQN：

| 专家 | 训练目标 | 激活条件 |
|-----------|-------------|-------------|
| 战斗 DQN | 射击、躲避 | 有敌人可见 |
| 探索 DQN | 找秘密、道具 | 无敌人在场 |
| Boss DQN | Cyberdemon 等 | Boss 战 |
| 导航 DQN | 寻路、找钥匙 | 迷宫关卡 |

微型门控网络（50KB）每帧选择专家。

**优点**：每个模型极小、可能超越单个 2MB 模型
**缺点**：训练复杂、切换可能突兀

## 方案 6：模仿学习 {#方案6-模仿学习}

不靠 RL，而是通过观察人类玩来学习。

### 6a. 行为克隆（BC）

```
人类玩 → 记录（帧, 动作）→ 监督学习
                         → CNN 预测人类动作
```

**数据需求**：~10,000 条演示（~5 小时游戏）
**训练**：一次性监督学习，无需 RL

**优点**：
- 训练快（小时级，非天级）
- 无需设计奖励函数
- 自然模仿人类风格
- 利用你自己的游戏数据

**缺点**：
- 只能达到演示者水平
- 无法超越人类
- 分布偏移：AI 进入人类从未到过的状态 → 失败

### 6b. BC → RL 微调

```
阶段 1：录制 5 小时你的 DOOM 游戏
阶段 2：从你的数据训练 BC 模型（PC 上 1 小时）
阶段 3：部署 BC 模型到手机
阶段 4：手机端 RL 从 BC 基线开始微调
        → 一开始就和你一样好，然后变得更好
```

**优点**：没有"蠢笨阶段"、AI 起步即有能力、持续进步
**缺点**：需要录制数据

## 方案 7：蒙特卡洛树搜索（MCTS）{#方案7-mcts}

MCTS 通过模拟可能的未来来提前规划，选出最有前景的路径。AlphaGo 使用的方法。

**缺点**：
- 需要快速前向模型（模拟 DOOM 物理 — 难以实现）
- 500 次模拟 × 3ms = 1.5 秒/决策（太慢）
- DOOM 动作太多（224 个）无法有效搜索
- 状态空间巨大且部分可观测

**结论**：❌ 不可行。没有学到的世界模型无法工作（回到 DreamerV3）。

| 战略推理 | 🟢 低 | 混合 DQN+VLM |

## 推荐方案与分阶段路线图 {#推荐方案}

### 阶段 1：基础（v1.0）— Double Dueling DQN，纯手机端

```
模型：Double DQN + Dueling 架构
训练：纯手机端 RL 从零开始
模式："训练模式"开关 — 让它通宵玩
模型保存：持久化权重、从断点恢复继续学习
```

**为什么**：简单、性能、以及"看它学习"的酷炫体验的最佳平衡。

### 阶段 2：引导启动（v1.1）— BC + RL

```
录制人类游戏 → 训练 BC 模型 → 部署为起点
手机端 RL 从人类基线开始微调
```

**为什么**：消除"蠢笨阶段" — AI 一开始就有能力并能不断提升。

### 阶段 3：战略（v2.0）— 混合 DQN + VLM

```
本地 DQN 处理实时战斗
云端/本地 VLM 提供偶尔的战略决策（每 5 秒）
```

**为什么**：在不牺牲实时性能的前提下增加高级智能。

### 阶段对比

| | v1.0 纯 RL | v1.1 BC+RL | v2.0 混合 DQN+VLM |
|---|---|---|---|
| **首次有趣的时间** | 1 小时 | 即时 | 1 小时 |
| **能力上限** | 高（持续提升） | 更高（从人类起步） | 最高（战略+战斗） |
| **离线可用** | ✅ 是 | ✅ 是 | ❌（云端 VLM） |
| **APK 大小影响** | +3.5 MB | +3.5 MB | +3.5 MB（+2GB 本地 VLM） |
| **实现难度** | ⭐⭐ 中 | ⭐⭐⭐ 中高 | ⭐⭐⭐⭐⭐ 高 |
| **电池影响** | 中（训练） | 中（训练） | 低（仅推理，如用云端 VLM） |
| **个性化** | 适应游戏 | 适应你的风格 | 适应游戏+战略 |
| **"蠢笨阶段"** | ❌ 30-60 分钟 | ✅ 无（起步即人类水平） | 取决于 DQN 引导方式 |
| **依赖** | 无 | PC 训练 BC（1 小时） | 网络 + API 密钥（云端 VLM） |
| **自我进化** | ✅ 持续 | ✅ 从人类基线向上 | ✅ 仅 DQN 部分 |
| **酷炫程度** | 🤯 "它自学的！" | 😎 "玩得跟我一样！" | 🧠 "它有个大脑！" |

### 阶段 4：专业化（v2.5）— 混合专家

```
多个微型专家处理战斗/探索/Boss
门控网络每帧选择专家
```

**为什么**：每 MB 最高性能 — 每个专家针对一个场景高度优化。

## 暂时不做的方案

| 方案 | 原因 |
|----------|--------|
| DreamerV3 | 太大太慢，等骁龙 NPU 成熟再说 |
| 纯 VLM 玩家 | 比实时 FPS 慢 100-400 倍，v2.0 仅做战略层 |
| MCTS | 需要学到的世界模型，与 DreamerV3 同样问题 |
| Rainbow DQN | v1.0 过度设计，逐步加入各组件 |
| 手机端 VLM | 2GB 模型装手机不现实，等模型压缩进步 |
